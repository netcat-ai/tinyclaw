TinyClaw 目标架构设计报告

本报告定义 TinyClaw 的目标架构。系统以 gRPC 写入消息、PostgreSQL 状态机、 主服务模型运行、sandbox 工具执行、jobs outbox 为核心边界。

日期：2026-05-13 范围：目标架构 约束：最佳实践优先

1. 目标

TinyClaw 的目标形态是一个消息驱动的 AI worker 服务。入口只负责写入事实， 调度层负责可靠处理，模型层在主服务内完成推理，sandbox 只执行需要隔离的工具。

  • 唯一正式入口：SubmitMessages gRPC。
  • 唯一入站事实源：messages。
  • 唯一处理驱动：pending message dispatcher。
  • 默认模型调用：主服务内 AgentRunner。
  • 隔离执行：sandbox ToolExecutor。
  • 唯一外发事实源：jobs outbox。
入口 RPC 不直接调用模型，不直接写 jobs，不绕过 messages 状态机。

2. 架构原则

事实先落库

任何输入先写入 messages，再由 dispatcher 异步处理。

单一状态机

消息只通过 pending -> processing -> done/failed 推进。

主服务持有推理循环

prompt、模型调用、tool loop、最终回复生成都在主服务内完成。

sandbox 最小职责

sandbox 只执行工具调用，不拥有会话调度、消息状态和外发状态。

4. 领域模型

4.1 messages

messages 保存所有入站事实和处理状态。它不保存模型中间推理， 也不保存工具调用明细。 

CREATE TABLE messages (
  id UUID PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  source TEXT NOT NULL,
  source_message_id TEXT NOT NULL,
  room_id TEXT NOT NULL,
  chat_type TEXT NOT NULL CHECK (chat_type IN ('direct', 'group')),
  from_id TEXT NOT NULL,
  from_name TEXT NOT NULL DEFAULT '',
  payload JSONB NOT NULL,
  status TEXT NOT NULL CHECK (
    status IN ('pending', 'processing', 'done', 'failed')
  ),
  available_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  locked_at TIMESTAMPTZ,
  attempt_count INTEGER NOT NULL DEFAULT 0,
  last_error TEXT NOT NULL DEFAULT '',
  msg_time TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (tenant_id, source, source_message_id)
);

4.2 rooms

rooms 保存 room 级元数据和运行策略，不承担消息游标。 

CREATE TABLE rooms (
  tenant_id TEXT NOT NULL,
  room_id TEXT NOT NULL,
  chat_type TEXT NOT NULL CHECK (chat_type IN ('direct', 'group')),
  title TEXT NOT NULL DEFAULT '',
  runner_policy JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (tenant_id, room_id)
);

4.3 jobs

jobs 是外发 outbox。只有最终需要发送给用户的回复进入 jobs。 

CREATE TABLE jobs (
  id UUID PRIMARY KEY,
  seq BIGSERIAL UNIQUE,
  tenant_id TEXT NOT NULL,
  bot_id TEXT NOT NULL,
  recipient_alias TEXT NOT NULL,
  message TEXT NOT NULL,
  source_message_ids UUID[] NOT NULL,
  idempotency_key TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (tenant_id, idempotency_key)
);

4.4 tool_calls

tool_calls 保存工具调用审计和重试边界。工具执行是独立状态机， 不污染 messages 状态。 

CREATE TABLE tool_calls (
  id UUID PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  room_id TEXT NOT NULL,
  message_batch_id UUID NOT NULL,
  tool_name TEXT NOT NULL,
  input JSONB NOT NULL,
  output JSONB,
  status TEXT NOT NULL CHECK (
    status IN ('pending', 'running', 'succeeded', 'failed')
  ),
  error TEXT NOT NULL DEFAULT '',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

5. gRPC 契约

5.1 Ingress API

service ClawmanIngress {
  rpc SubmitMessages(SubmitMessagesRequest) returns (SubmitMessagesResponse);
}
message SubmitMessagesRequest {
  string tenant_id = 1;
  repeated InboundMessage messages = 2;
}

message InboundMessage {
  string source = 1;
  string source_message_id = 2;
  string room_id = 3;
  string chat_type = 4;
  string from_id = 5;
  string from_name = 6;
  string msg_time = 7;
  bytes payload_json = 8;
}

message SubmitMessagesResponse {
  int32 accepted = 1;
  int32 duplicated = 2;
  repeated string message_ids = 3;
}

5.2 Tool Runtime API

service ToolRuntime {
  rpc ExecuteTool(ExecuteToolRequest) returns (ExecuteToolResponse);
}

message ExecuteToolRequest {
  string tenant_id = 1;
  string room_id = 2;
  string call_id = 3;
  string tool_name = 4;
  bytes input_json = 5;
}

message ExecuteToolResponse {
  string call_id = 1;
  bytes output_json = 2;
}
工具执行使用 unary RPC。需要流式输出时再单独引入 streaming tool API， 不把普通工具调用一开始就设计成双向流。

6. 处理流程

  1. Producer 调用 SubmitMessages。
  2. Ingress 校验租户、来源、room、payload，并按幂等键写入 messages。
  3. Dispatcher 原子 claim 一组 pending 消息，状态改为 processing。
  4. Dispatcher 读取 room metadata 和当前处理批次上下文。
  5. AgentRunner 构造 prompt 并调用模型。
  6. 模型请求 tool 时，AgentRunner 调用 ToolExecutor。
  7. ToolExecutor 确保 sandbox 存在，执行工具并返回结构化结果。
  8. AgentRunner 继续模型循环，直到得到最终回复。
  9. 服务以幂等键写入 jobs。
  10. 写入 jobs 成功后，消息状态改为 done。

7. 状态机

状态 含义 下一步
pending 已入库，等待处理 processing
processing 已被 worker claim，正在运行模型或工具 done / failed
done 最终回复已写入 jobs 终态
failed 超过重试上限或不可恢复错误 人工处理或重新投递

无效消息由 ingress 直接拒绝，不写入 messages。worker claim 使用数据库事务和 FOR UPDATE SKIP LOCKED，避免多 worker 重复处理同一批消息。

8. 核心接口

type MessageIngress interface {
    Submit(ctx context.Context, req SubmitMessagesRequest) (SubmitMessagesResult, error)
}

type AgentRunner interface {
    Run(ctx context.Context, req AgentRequest) (AgentResult, error)
}

type ToolExecutor interface {
    Execute(ctx context.Context, req ToolRequest) (ToolResult, error)
}

type JobOutbox interface {
    Enqueue(ctx context.Context, job Job) (Job, error)
}

这些接口是模块边界。每个接口只表达一个业务职责。

9. 部署形态

组件 职责 伸缩方式
clawman-api gRPC ingress、Android jobs API、metrics 水平扩容，无本地状态
clawman-worker dispatcher、AgentRunner、ToolExecutor client 水平扩容，数据库锁协调
tool-sandbox 隔离工具执行 按 room 或 tool call 创建，空闲回收
postgresql messages、jobs、rooms、tool_calls 托管数据库或平台层服务

11. 实施顺序

  1. 按目标 schema 重建 messages、jobs、rooms、tool_calls。
  2. 实现 ClawmanIngress.SubmitMessages。
  3. 实现数据库 claim worker：pending -> processing。
  4. 实现主服务 AgentRunner。
  5. 实现幂等 jobs outbox。
  6. 实现 sandbox ToolRuntime.ExecuteTool。
  7. 把 tool loop 接入 AgentRunner。
  8. 补 metrics、structured logs、失败重试和 idle sandbox 回收。

12. 验收标准

  1. gRPC 提交消息后，消息以幂等方式写入 messages。
  2. worker 可并发运行，且不会重复 claim 同一条消息。
  3. 普通文本消息不启动 sandbox，也能生成回复并写入 jobs。
  4. 需要工具时才启动 sandbox，并记录 tool_calls。
  5. 模型失败、工具失败、jobs 写入失败都有明确状态和重试策略。
  6. jobs 使用幂等键，重复处理不会重复外发。
  7. 所有关键路径都有 metrics 和结构化日志。
